# Cassini AI API Structure

This project does not expose an HTTP REST API yet. The current public interface is a
Python CLI plus JSON-compatible data contracts produced by the monitoring package.
This file documents those contracts so a future web API can wrap them consistently.

## Runtime Entry Point

Command:

```text
python hourly_monitor.py [options]
```

Main functions:

- `hourly_monitor.parse_args()`: parses CLI options.
- `hourly_monitor.build_report(args)`: builds the in-memory report dictionary.
- `hourly_monitor.main()`: runs the monitor and writes outputs.

Important CLI options:

- `--config`: path to monitor configuration JSON. Default: `config/monitor_config.json`.
- `--sensors`: path to sensor CSV. Default: `data/sensors.csv`.
- `--demo`: use bundled demo weather values.
- `--no-context`: skip historical weather and elevation context.
- `--no-copernicus`: skip Copernicus satellite catalogue checks.
- `--no-predictions`: skip multi-day predictions.
- `--prediction-days`: number of Open-Meteo daily prediction rows.
- `--update-resources`: refresh cached OSM resource files for the submitted area.
- `--land-cover`: optional classified satellite land-cover input (`.tif`, `.geojson`, `.json`, `.csv`).
- `--land-cover-cache`: per-bbox cache directory for classified satellite land-cover files.
- `--land-cover-keep-latest`: number of cached classified files to keep per analyzed bbox.
- `--no-land-cover-download`: skip automatic Copernicus CLMS LCFM download when cache is empty.
- `--land-cover-max-pixels`: maximum output pixels for one automatic Copernicus CLMS raster request.
- `--land-cover-year`: Copernicus CLMS LCFM LCM-10 reference year to request. Defaults to the current UTC year.
- `--print-json`: print the generated report to stdout.

## Configuration Contract

File: `config/monitor_config.json`

Top-level fields:

```text
area:
  name: string
  polygon: list[[longitude, latitude]]
weather:
  forecast_window_hours: integer
thresholds:
  flood_precip_high_m: number
  drought_precip_low_m: number
  drought_temp_hot_c: number
  storm_wind_high_ms: number
  heat_high_c: number
  wildfire_temp_hot_c: number
  wildfire_wind_high_ms: number
sensor_health:
  min_online_ratio: number
  stale_after_minutes: number
context: object
copernicus: object
```

## Input Data Contracts

Sensor CSV: `data/sensors.csv`

```csv
sensor_id,type,latitude,longitude,status,last_seen_utc
```

Weather is fetched from Open-Meteo for the configured polygon or runtime `--bbox`.
`--demo` uses bundled in-code sample values for smoke tests only.

## Report Output Contract

`build_report(args)` returns a dictionary shaped like this:

```json
{
  "timestamp": "2026-04-25T00:00:00Z",
  "area": "Iasi",
  "weather": {},
  "calamities": {},
  "sensors": {
    "total": 0,
    "online": 0,
    "offline": 0,
    "stale": 0,
    "working": false
  },
  "working": false,
  "notes": "text",
  "copernicus": {},
  "context": {},
  "predictions": {},
  "resources": {},
  "regional_calibration": {},
  "satellite_land_cover": {}
}
```

Always present:

- `timestamp`
- `area`
- `weather`
- `calamities`
- `sensors`
- `working`
- `notes`

Conditionally present:

- `copernicus`: omitted when `--no-copernicus` is used or the check is skipped.
- `context`: omitted when `--no-context` is used.
- `predictions`: omitted unless provider is `openmeteo`, context is enabled, and predictions are enabled.
- `resources`: omitted when `--no-resources` is used.
- `regional_calibration`: broad bbox-based threshold calibration used for scoring.
- `satellite_land_cover`: final dashboard-safe land-cover/crop summary. When `--land-cover` is omitted, the monitor searches the newest supported Copernicus/classified file in the per-bbox land-cover cache, then requests a Copernicus CLMS LCFM LCM-10 raster through Sentinel Hub Process API when credentials are configured and the bbox fits `--land-cover-max-pixels`. Crop type percentages require a crop-specific classified layer/table.

## Calamity Score Contract

`calamities` contains one object per hazard:

```text
flood
drought
storm
heatwave
wildfire
```

Each hazard object includes:

```json
{
  "score": 0.0,
  "risk_percent": 0.0,
  "risk_index_percent": 0.0,
  "is_probability": false,
  "risk": "low",
  "title": "Flood",
  "explanation": "text",
  "input_values": {},
  "normal_limits": {},
  "risk_scale": {
    "none": "0-4.99%: no meaningful signal",
    "extremely_low": "5-14.99%: extremely low risk",
    "very_low": "15-29.99%: very low risk",
    "low": "30-44.99%: low risk",
    "moderate": "45-59.99%: moderate risk",
    "high": "60-79.99%: high risk",
    "extreme": "80-100%: extreme risk"
  }
}
```

Risk labels:

- `none`: score below 0.05.
- `extremely_low`: score from 0.05 to below 0.15.
- `very_low`: score from 0.15 to below 0.30.
- `low`: score from 0.30 to below 0.45.
- `moderate`: score from 0.45 to below 0.60.
- `high`: score from 0.60 to below 0.80.
- `extreme`: score from 0.80 to 1.0.

Important note: `risk_index_percent` is a normalized warning index, not an
official alert and not an event probability.

## Suggested Future HTTP API Wrapper

A minimal REST wrapper can map the current contracts as:

```text
GET  /health
  Returns service status and dependency availability.

GET  /config
  Returns sanitized monitor configuration.

POST /reports
  Body: optional provider and runtime flags.
  Runs build_report(...) and returns the generated report.

GET  /reports/latest
  Returns the latest generated dashboard payload.

GET  /reports/history
  Query: limit, since, hazard.
  Requires a database or external storage because append-only local reports are no longer written.

POST /weather/score
  Body: WeatherFeatures JSON.
  Returns calamity scores without writing files.
```

For the first implementation, `POST /reports` should call the same
`build_report(args)` path used by the CLI to avoid maintaining two separate
calculation flows.
